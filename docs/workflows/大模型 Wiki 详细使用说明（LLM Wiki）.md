# LLM Wiki 详细使用说明

这份说明面向这个 Obsidian vault「脑子」的实际使用，不是抽象概念介绍。

一句话：
LLM Wiki 不是“有问题时让模型搜一下资料”，而是“把资料、理解、判断、问题、导航持续沉淀成一个会长大的知识库”。

---

## 1. 它到底是干什么的

传统聊天的问题是：
- 当下回答可能不错
- 但下一次上下文一断，很多高价值内容就丢了
- 同一件事容易反复解释、反复整理

LLM Wiki 的目标是把这些高价值内容留下来，并且留下的是“可继续生长的页面”，不是零散聊天记录。

在这个系统里：
- Obsidian 是知识库界面
- Markdown 页面是长期记忆单元
- LLM 负责整理、链接、归纳、更新
- 你负责给来源、判断价值、决定继续深挖什么

所以它更像“持续维护一个脑子”，不是“临时问答”。

---

## 2. 这个 vault 的核心结构

### 01 来源
放原始来源。
适合：
- X 帖子
- 文章
- 论文
- 视频整理
- GitHub 项目介绍
- 网页资料

作用：
- 保留事实来源
- 写摘要
- 提炼关键要点
- 给后续 Topic / Synthesis 提供依据

### 02 实体
放稳定对象。
适合：
- 人物
- 公司
- 产品
- 工具
- 组织
- 概念

作用：
- 给一个对象建立长期档案
- 逐步积累它的定义、特点、关联页面

### 03 主题
放会持续增长的主题。
适合：
- 一个问题域
- 一类方法论
- 一个长期研究方向

作用：
- 把多条来源串起来
- 形成结构化理解
- 记录“当前理解”和“未解决问题”

### 04 综合
放综合判断。
适合：
- 对比
- 结论
- 推荐
- 分层理解
- 方法论提炼

作用：
- 输出“你现在的判断”
- 明确依据、保留意见和下一步验证方向

### 06 主张与问题
放主张、问题、矛盾。
适合：
- 一个可验证判断
- 一个重要开放问题
- 两个来源之间的冲突

作用：
- 避免所有内容都糊成“笔记”
- 把可验证对象单独拎出来

### 07 地图
放导航页。
适合：
- 主题索引
- 入口页
- 专题地图
- 助手协作规则

作用：
- 帮你快速找到知识入口
- 帮不同模型/不同上下文快速恢复工作状态

### 08 工作流
放操作说明和流程。
适合：
- 怎么处理新来源
- 怎么更新页面
- 怎么做长期维护

作用：
- 把“怎么用这个脑子”写清楚
- 降低换模型、换上下文后的损耗

## 3. LLM Wiki 和普通笔记最大的区别

普通笔记常见问题：
- 看到一个东西就记一条
- 没有分类
- 没有链接
- 没有更新机制
- 过几天就找不到、也不想看

LLM Wiki 的关键差异：

### 3.1 它是分层的
不是所有内容都放一页。
最常见的分层是：
- Source：原始来源
- Topic：主题理解
- Synthesis：你的判断
- Map：导航入口

### 3.2 它会回写
一次有价值的聊天，不应该只留在对话里。
只要结论有复用价值，就应该回写进某个页面。

### 3.3 它强调链接
不是记完就结束，而是：
- 这个来源关联谁？
- 属于哪个主题？
- 改变了哪个判断？
- 是否要进入某个地图页？

### 3.4 它允许不确定性
不是所有东西都马上变成结论。
可以把内容拆成：
- 已确认的事实
- 当前判断
- 保留意见
- 开放问题
- 互相矛盾的说法

这比“假装全懂了”更有用。

## 4. 你平时到底怎么用

最简单的实际用法只有四类。

### 用法 A：存一个来源
你给我：
- 一个链接
- 一段文字
- 一张图或一个文件
再说一句：存笔记

我默认会做：
1. 提取内容
2. 判断它属于哪类来源
3. 写到 `01 来源/`
4. 如果主题明显，会顺手更新 `03 主题/`
5. 如果它值得长期导航，会更新 `07 地图/`

适合：
- X 帖子
- GitHub 项目
- 论文
- 文章

### 用法 B：研究一个主题
你给我一句：
- 研究一下 XX
- 帮我整理 XX
- 这个方向值不值得继续

我默认会做：
1. 先找脑子里已有内容
2. 再补外部资料
3. 更新或新建 Topic
4. 如果已有判断，就写 Synthesis
5. 必要时补 Map

适合：
- AI 短剧
- Persona Distillation
- LLM Wiki
- 社群方法论

### 用法 C：要结论，不只是资料
你给我一句：
- 帮我判断
- 给我一个建议
- 做个对比
- 值不值得试

我默认会做：
1. 引用来源
2. 抽出评价维度
3. 输出当前结论
4. 写进 `04 综合/`
5. 给出置信度和保留意见

### 用法 D：维护脑子本身
你给我一句：
- 整理一下脑子
- 把这个主题串起来
- 做个地图页
- 检查哪里重复了

我默认会做：
1. 找重复页、碎片页
2. 合并链接关系
3. 补导航页
4. 强化入口页
5. 必要时把零散来源提升成 Topic / Synthesis

## 5. 什么时候该放哪一层

这是最关键的判断规则。

### 放 Source，当：
- 它主要是一个外部来源
- 你现在先想保存材料
- 还没形成明确判断

例子：
- 一条 X 长帖
- 一篇论文
- 一个项目 README

### 放 Entity，当：
- 这是一个稳定对象
- 以后大概率还会反复提到
- 你需要一个“这个东西到底是什么”的固定页面

例子：
- Obsidian
- LLM Wiki
- Andrej Karpathy
- 某个长期关注的工具

### 放 Topic，当：
- 这个方向会持续增长
- 多条来源都指向同一个主题
- 你不想让理解散在很多来源页里

例子：
- Persona Distillation
- AI 短剧工具
- 门派型组织与合流体

### 放 Synthesis，当：
- 你已经不只是记录，而是在判断
- 你要对比、推荐、分层、下结论
- 这份内容以后会被反复引用

例子：
- 哪几个项目最值得试
- LLM Wiki 为什么和 RAG 不一样
- 某套方法论值不值得学

### 放 Claim / Question / Contradiction，当：
- 某个说法值得单独验证
- 某个问题需要持续追踪
- 两个来源互相打架

### 放 Map，当：
- 这个方向已经有一批页面了
- 你需要一个“从哪开始看”的入口
- 后续可能换模型、换上下文继续做

## 6. 你怎么给我指令最省事

你不需要每次都讲很完整，下面这些话就够用。

### 6.1 单纯存资料
- 存笔记
- 这个存一下
- 把这个放进脑子

### 6.2 要我顺手做结构化整理
- 存笔记，严格按 LLM Wiki 做
- 存一下，顺手建主题页
- 存一下，看看要不要补地图

### 6.3 要研究
- 研究一下这个方向
- 把这个主题梳理出来
- 结合脑子里现有内容继续整理

### 6.4 要判断
- 给我一个结论
- 值不值得继续做
- 做个对比
- 最后给建议

### 6.5 要维护知识库
- 检查这个主题是不是散了
- 把这些笔记串起来
- 做一个入口页
- 清理重复和碎片

你现在已经明确说了“严格执行”，所以后面我会默认按更严格版本处理：
- 不只存来源
- 尽量补 Topic
- 有结论就补 Synthesis
- 需要入口就补 Map

## 7. 一个标准工作流长什么样

下面是最常见的完整流程。

### 场景：你发来一个链接并说“存笔记”

我会按下面这条链走：
1. 抓取原内容
2. 判断来源类型
3. 新建 Source note
4. 看它涉及哪些实体
5. 看它属于哪个 Topic
6. 如果它改变了已有判断，更新 Synthesis
7. 如果这个方向开始成体系，更新 Map
8. 最后告诉你存到了哪里、顺手补了什么

### 场景：你说“研究一下 AI 短剧”

我会按下面这条链走：
1. 先查脑子里已有 AI 短剧相关页面
2. 读相关来源、主题、综合页
3. 必要时补抓新的外部资料
4. 更新 Topic
5. 产出一篇 Synthesis
6. 如果内容多了，做地图页或更新导航

### 场景：你说“给我一个推荐”

我会按下面这条链走：
1. 先看现有知识库
2. 补来源
3. 比较对象
4. 输出判断
5. 把判断沉淀到 Synthesis
6. 把证据挂回相关来源和主题

## 8. 好笔记长什么样

### 好的 Source note
应该有：
- 来源信息
- 中文摘要
- 原文整理 / 关键内容
- 关键信息
- 指向相关 Topic / Synthesis 的链接

### 好的 Topic note
应该有：
- 核心问题
- 子主题
- 相关来源
- 当前理解
- 下一步问题

### 好的 Synthesis note
应该有：
- 核心结论
- 支持证据
- 保留意见
- 当前置信度
- 下一步验证
- 来源链接

### 好的 Map page
应该有：
- 这个领域的入口说明
- 推荐阅读顺序
- 相关页面分组
- 当前开放问题
- 后续扩展方向

---

## 9. 什么情况下要升级页面

### 从 Source 升级到 Topic
当你发现：
- 连续几条资料都在讲同一个东西
- 同一主题开始反复出现
- 已经不适合只靠来源页理解

### 从 Topic 升级到 Map
当你发现：
- 一个主题下面已经挂了很多来源和综合页
- 你想给以后留一个“从这里开始”的入口
- 你希望换模型后也能快速接续

### 从聊天升级到 Synthesis
当你发现：
- 聊天里已经形成稳定判断
- 这份判断以后还会再用
- 它不该只留在对话记录里

## 10. 日常维护建议

### 每次有新资料时
最少做这三件事：
1. 存 Source
2. 链接到已有 Topic
3. 看要不要更新已有 Synthesis

### 每周可做一次小维护
- 看 `00 收件箱/` 有没有没处理的内容
- 看有没有重复主题
- 看 Topic 有没有长得太散
- 看哪些 Synthesis 已经过时
- 看 Maps 还能不能更好导航

### 每月可做一次结构维护
- 哪些主题值得单独做地图页
- 哪些实体页需要补全
- 哪些开放问题已经有答案
- 哪些矛盾需要单列出来

## 11. 常见错误

### 错误 1：什么都只存来源
后果：
资料越来越多，但理解没有沉淀。

### 错误 2：什么都写成一大篇综合
后果：
来源、事实、判断混在一起，后面很难维护。

### 错误 3：重复建很多差不多的页面
后果：
你会越来越不知道该更新哪一页。

### 错误 4：没有导航页
后果：
明明存了很多东西，但下次进来还是不知道从哪看。

### 错误 5：高价值聊天结论不回写
后果：
每次换上下文都要重来一遍。

### 错误 6：把临时任务进度当知识存
后果：
知识库会被很多短期噪音污染。
