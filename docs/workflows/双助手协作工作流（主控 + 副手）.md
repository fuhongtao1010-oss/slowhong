# 双助手协作工作流（主控 + 副手）

## 目的
在主助手额度较贵或有额度限制时，把重复、长流程、机械整理类工作分流给副手模型；主助手只负责拆任务、验收、纠偏和最终交付。

## 当前落地配置
- 主助手：当前对话里的主控助手
- 副手（简单消息分流）：`minimax-cn / MiniMax-M2.5`
- 副手（子任务 delegation）：`minimax-cn / MiniMax-M2.7`
- 当前 Hermes 配置位置：`/home/hong/.hermes/config.yaml`

对应配置思路：
1. 短、简单、低风险的消息，优先走便宜模型。
2. 需要独立执行的子任务，优先交给副手模型完成。
3. 主助手保留最终判断权，不把副手输出直接当最终结果。

## 分工规则
### 主助手负责
- 理解用户真实目标
- 拆任务、定标准、定验收口径
- 本地环境操作与高风险动作判断
- 复杂排障、关键取舍、最终定稿
- 对副手结果做验收、补漏、返工
- 明确给用户回执：已完成/未完成、做了什么、结果在哪

### 副手负责
- 长文初译
- 批量摘要、归类、标签化
- 大量资料初步整理
- 按模板生成初稿
- 重复性较高的清洗、改写、信息提取
- 可明确约束边界的独立子任务

## 默认工作链路
用户提出需求
→ 主助手判断是否适合外包
→ 适合的部分交给副手
→ 主助手验收副手结果
→ 必要时返工或升级模型
→ 主助手交付最终版本

## 哪些任务优先交给副手
- 翻译英文资料为中文初稿
- 文章/网页抓取后的初步整理
- 批量提炼要点
- 结构化表格草稿
- Obsidian 笔记初稿
- 代码说明、接口说明、注释初稿

## 哪些任务不要直接丢给副手
- 需要直接改系统、删文件、装环境的关键动作
- 根因不明的复杂排障
- 最终结论必须非常稳的判断题
- 用户很在意“不要降级”的正式交付
- 需要强工具协同和连续上下文把控的复杂任务

## 升级与降级规则
### 可降级给副手
- 初稿
- 草稿翻译
- 第一轮分类
- 第一轮信息抽取

### 不可偷偷降级
- 用户明确要求高质量终稿
- 用户明确要求我亲自判断
- 风险高、影响大的本地操作

如果只能给近似版，必须先说清楚这是降级方案，不能装作已经完全按高规格完成。

## 对未来助手的要求
- 不要让用户同时盯多个助手；默认仍只对接一个主入口。
- 副手是执行员，不是最终拍板者。
- 副手结果必须经过主助手二次检查，尤其是：
  - 事实是否漏项
  - 是否误解任务
  - 是否把“初稿”冒充“终稿”
  - 是否遗漏全文、真实链接、关键上下文
- 对外给用户的最终回执，必须由主助手统一给出。

## 当前配置含义
### 1. 简单消息分流
适合：
- 很短的问答
- 低复杂度寒暄/确认
- 不涉及工具和长上下文的小请求

### 2. delegation 子助手
适合：
- 独立研究子任务
- 需要隔离上下文的子工作
- 不想把中间过程全塞进主对话的任务
